# R28 — Live history graphs in the web UI

**Requirement.** The web UI shall display rolling 1-minute history graphs of the
Art-Net packet rate and the DMX refresh rate, built client-side from periodic
`/api/state` polls, with no data logging on the ESP.

## Rationale
The web UI (R23) shows instantaneous live state only. A short rolling history of
the two key health metrics — how much Art-Net traffic is arriving and the actual
DMX frame rate — makes drop-outs and jitter visible at a glance during a show.
Traces to **G6** (field-serviceable: monitor the node over the network).

## Design notes
- **No on-device logging.** The browser keeps the history; the firmware only
  exposes the two current values. This keeps the ESP side trivial and stateless.
- **Art-Net packets/sec counter (firmware, `eth_task`).** Increment a counter on
  every datagram that passes the `"Art-Net"` header check (all opcodes, before the
  per-opcode handling). Track the time since the last calculation; when it exceeds
  1 s, store `counter` into the published `artnet_pps` variable and reset both the
  counter and the timer. Computed inline on the receive path — no extra task.
  - **Decays to 0 during silence.** The socket has a 250 ms `SO_RCVTIMEO`, so
    `recvfrom` wakes even with no traffic and the 1 s window still fires, dropping
    the published rate to 0 when Art-Net stops. This is what makes the graph useful
    for "are packets even reaching the node?" while debugging a rig. (The timeout
    makes `recvfrom` return `-1` on each idle tick, which required signed-length
    handling — closing bug B7. See [reqs/B7.md](reqs/B7.md).)
- **Exposure.** `artnet_pps` is added to `/api/state` as `"pps"`.
- **Client.** `index.html` polls `/api/state` ~once per second, keeps the last 60
  samples of `pps` and `refresh` in two arrays, and redraws two lightweight
  `<canvas>` sparklines (no libraries — R7/R27). Newest sample at the right edge;
  auto-scaled to the window max with a 0 baseline.

## Acceptance criteria
- [ ] `eth_task` counts every Art-Net packet and republishes packets/sec ~every 1 s.
- [ ] `/api/state` includes a numeric `pps` field.
- [ ] The page shows two graphs spanning the last ~60 s, updating live.
- [ ] History is entirely client-side; nothing is stored on the ESP.
- [ ] No external libraries/CDNs; page stays self-contained and small (R7/R27).
