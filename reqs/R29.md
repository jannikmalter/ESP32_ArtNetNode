# R29 — Core-0 CPU utilization metric

**Requirement.** The node shall measure core-0 CPU utilization and report it,
shown in the web UI as a live value and a rolling 1-minute history graph.

## Rationale
Core 0 runs the variable load — `eth_task` (Art-Net ingest), the HTTP server, and
`stats_task` — while core 1 is reserved for DMX. Knowing how much headroom core 0
has helps judge whether the node is near saturation under heavy Art-Net traffic.
Traces to **G2** (reliability/performance); surfaced through the web UI (builds on
R23/R28).

## Design notes
- **Method: FreeRTOS run-time stats.** Enable
  `CONFIG_FREERTOS_GENERATE_RUN_TIME_STATS` (esp_timer clock source). A dedicated
  `stats_task` on core 0 samples core-0's idle task run-time counter
  (`ulTaskGetRunTimeCounter(xTaskGetIdleTaskHandleForCore(0))`) over a 1 s wall
  window and computes `load0 = 100 * (1 - idle_busy / wall)`. No calibration; the
  run-time counter and the wall clock share the same esp_timer µs base. Counter
  deltas use modular unsigned subtraction so a 32-bit counter wrap is harmless.
- **Core 1 is not measured (by design).** `dmx_task` holds core 1 in
  `vPortEnterCritical()` with the scheduler/interrupts off forever (R9/R10), so its
  idle task never runs and any scheduler-based figure is meaningless — it is 100%
  `dmx_task` by definition.
- **Overhead.** Run-time stats add a couple of esp_timer reads per context switch
  on core 0; the cycle-counted DMX core is unaffected (it bypasses the scheduler).
- **Exposure.** `load0` (integer percent) added to `/api/state`; the page keeps 60
  samples and draws a third sparkline (fixed 0-100 scale), no libraries (R7/R27).

## Acceptance criteria
- [ ] `stats_task` publishes a 0-100 core-0 load figure ~every 1 s.
- [ ] `/api/state` includes a numeric `load0` field.
- [ ] The web UI shows a live value and a 1-minute graph for core-0 load.
- [ ] DMX output timing is unaffected by enabling run-time stats.
- [ ] No external libraries; page stays self-contained (R7/R27).
