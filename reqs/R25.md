# R25 — ArduinoOTA/espota-compatible OTA (optional, "PlatformIO Upload button")

**Requirement.** Where the protocol proves tractable to implement, the node shall
also accept OTA firmware update over the ArduinoOTA/espota protocol, so that
PlatformIO's native `upload_protocol = espota` flashes it directly from VSCode /
the PlatformIO toolbar — no bespoke host program.

## Rationale
The "dream" UX is clicking Upload in VSCode and having it flash over Ethernet.
PlatformIO already ships the host side (`espota.py`, the ArduinoOTA wire protocol);
the only missing piece is a device-side responder. Since we're on `framework =
espidf` (no ArduinoOTA library), we'd implement that responder ourselves on top of
the same OTA engine as [R26](R26.md). This is a **Could** — nice-to-have, gated on
the protocol being reasonable to reverse-engineer. Traces to **G6**, ESP-IDF only
(**R7**).

## Feasibility spike first (todo T12)
Before committing, read and assess:
- PlatformIO's bundled `espota.py` (Espressif/Arduino-ESP32) — the canonical
  ArduinoOTA wire format: UDP invitation on **port 3232**, optional MD5 auth, then
  a TCP image transfer (device connects back to the host).
- ESPHome's `espota2.py` + `esphome/components/ota` as a reference for a clean
  IDF-side implementation (its OTA is framework-independent over `esp_ota_*`).

Decide go/no-go from the spike. If the handshake is small and stable, implement it;
if it's gnarly or auth-heavy, drop R25 and rely on R26 (web upload) alone.

## Acceptance criteria
Only if the spike says "go":

- [ ] `pio run -e olimex-rack -t upload --upload-port <ip>` (with
  `upload_protocol = espota`) flashes the running node over Ethernet.
- [ ] Reuses the R26 OTA engine (inactive slot, validate, rollback-safe).
- [ ] Responder runs on core 0; live DMX on core 1 is undisturbed until reboot.
- [ ] Implemented with ESP-IDF + LwIP sockets only — no external libraries (R7),
  no Arduino framework.

## Design notes
- Device side: UDP listener on 3232 parses the invitation (command, host TCP port,
  size, MD5), optional MD5 challenge-response, then TCP-connects to the host and
  streams the image into `esp_ota_write`.
- Add `upload_protocol = espota` (+ optional `upload_flags = --auth=...`) per env in
  [platformio.ini](../platformio.ini) once the responder exists.
- **Security note:** this is an unauthenticated-by-default firmware-flash port on a
  live-lighting device. Fine on a trusted lighting LAN; require a password if it
  could ever face an untrusted network.
