# R26 — OTA firmware update via the web UI (primary channel)

**Requirement.** The node shall support OTA firmware update by uploading a new
image through the web UI: the browser streams a firmware file to the node, which
flashes and reboots into it. This is the **primary, committed OTA channel**.

## Rationale
We chose to stay on ESP-IDF and make browser upload the main update path — no
bespoke host program needed, and it reuses the web server that already exists for
config ([R23](R23.md)). Traces to **G6**. ESP-IDF only (**R7**). The
ArduinoOTA/espota-compatible path ([R25](R25.md)) is an optional extra on top of
the same engine.

## OTA engine (shared, this is the bulk of the work — todo T10)
Both R26 and R25 sit on one OTA core:
- **Partition layout change.** The firmware is currently single-app (see
  [info.md](../info.md)). OTA needs two app slots (`ota_0`/`ota_1`) + `otadata`.
  With flash pinned at "2MB" and the app at ~38% of a 1 MB partition, two app
  slots + otadata + NVS won't fit — almost certainly move to the chip's real 4 MB
  (the `esp32-poe` manifest already declares 4 MB). Decide and commit a new
  partition CSV.
- **Write path:** `esp_ota_begin/write/end` to the inactive slot,
  `esp_ota_set_boot_partition`, reboot. Enable rollback
  (`CONFIG_BOOTLOADER_APP_ROLLBACK_ENABLE`) and mark the new image valid only after
  a self-check, so a bad image reverts.
- **Off the DMX core.** The whole OTA path runs on core 0; core 1 keeps bit-banging
  DMX until the final reboot. Never touch DMX state mid-update.

## Acceptance criteria
All must hold before R26 is marked Done in `reqs.md`.

- [x] The web UI offers a firmware-upload control that streams an image to the node.
  ([src/index.html](../src/index.html) file input + `ota()`; raw octet-stream body.)
- [x] The node writes to the inactive OTA slot, validates, and boots it; a
  bad/aborted image rolls back (no brick). (`ota_post_handler`:
  `esp_ota_get_next_update_partition` -> `begin`/`write`/`end` ->
  `set_boot_partition` -> reboot; `esp_ota_end` validates; `app_main`
  rollback-confirm reverts an image that fails to come up.)
- [x] Update progress and success/failure are shown in the UI before the reboot.
  (XHR upload-progress bar + result message.)
- [x] OTA never disturbs live DMX timing on core 1. (Handler runs `stopDMX()`
  before the first flash op — DMX is fully stopped for the whole update, by
  decision — and only `startDMX()`s again on an error path; the success path
  reboots.)
- [x] Served and handled by `esp_http_server` only — no external libraries (R7).

**Decision (2026-06-26):** DMX is stopped for the entire update, not kept live
mid-flash. Updates happen between shows, so a few seconds of no output is
acceptable and it keeps the flash writes trivially safe against the core-1
critical section. Not yet hardware-verified (Done = implemented per project
convention).

## Design notes
- A multipart/octet-stream POST handler on the existing `esp_http_server` instance
  feeds the request body straight into `esp_ota_write`.

## Open questions

- Image authentication: plain, checksum-only, or signed? (secure boot is heavier).
